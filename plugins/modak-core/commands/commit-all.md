# 구조화 커밋 생성

변경사항을 분석하여 논리적 단위로 그룹화하고, 각 그룹별로 개별 커밋을 생성한다.

## 프로세스
1. `git status`와 `git diff`로 변경사항 파악
2. 변경된 파일들을 논리적 단위로 그룹화
3. 그룹화 계획을 사용자에게 보여주고 확인
4. 각 그룹별로 개별 커밋 생성

## 파일 그룹화 기준
- **같은 기능**: 하나의 기능을 위해 수정된 파일들
- **같은 모듈/디렉토리**: 같은 폴더 내 관련 파일들
- **같은 목적**: 버그 수정, 리팩토링, 설정 변경 등 목적이 같은 파일들
- **의존성**: A 파일이 B 파일을 import하면 함께 커밋

## 커밋 타입

| 타입 | 설명 |
|------|------|
| feat | 새로운 기능 |
| fix | 버그 수정 |
| chore | 설정 변경, 라이브러리 등 잡일 |
| refactor | 동작 변경 없이 코드 정리 |
| style | 포맷팅, 세미콜론 등 |
| docs | 문서 작성/수정 |
| test | 테스트 코드 |
| ci | CI/CD 설정 |
| perf | 성능 개선 |

## 커밋 포맷
```
<type>: <description>

- 세부설명1
- 세부설명2
```

## 규칙
- 한글 커밋 메시지 사용
- 첫 글자 소문자
- 마침표 없이 끝내기
- 명령형으로 작성 (추가, 수정, 삭제)
- 하나의 커밋 = 하나의 논리적 변경
- 관련 없는 변경사항은 절대 하나의 커밋에 섞지 않기
- 커밋하기 전에 그룹화 계획을 먼저 보여주고 확인받기
