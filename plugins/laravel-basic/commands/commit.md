# 커밋 생성

## 프로세스
1. git status와 git diff로 변경사항 파악
2. 변경된 파일들을 논리적 단위로 그룹화
3. 각 그룹별로 개별 커밋 생성
4. 관련 없는 변경사항은 절대 하나의 커밋에 섞지 않기

## 파일 그룹화 기준
- **같은 기능**: 하나의 기능을 위해 수정된 파일들
- **같은 모듈/디렉토리**: 같은 폴더 내 관련 파일들
- **같은 목적**: 버그 수정, 리팩토링, 설정 변경 등 목적이 같은 파일들
- **의존성**: A 파일이 B 파일을 import하면 함께 커밋

## 커밋 타입

| 타입 | 설명 |
|------|------|
| feat | 새로운 기능이 생겼을 때 |
| fix | 고장 난 걸 고쳤을 때 |
| chore | 설정 변경, 라이브러리 설치 등 잡다한 일 |
| refactor | 결과는 같은데 코드만 깔끔하게 정리했을 때 |
| style | 줄바꿈, 띄어쓰기 등 디자인적 코드 정리 |
| docs | 설명서(README) 등을 썼을 때 |
| test | 테스트 코드를 짰을 때 |

## 커밋 포맷
```
<type>: <description>

- 세부설명1
- 세부설명2
```

## 예시
```
feat: CRC-16 체크섬 검증 로직 추가

- ble_service.dart에 CRC 계산 함수 구현
- 패킷 수신 시 체크섬 검증 추가
```
```
fix: 0x96 명령어 수신 오류 수정

- 바이트 파싱 순서 수정
- 타임아웃 처리 로직 보완
```
```
chore: flutter 패키지 업데이트

- flutter_blue_plus 1.5.0 -> 1.6.0
- provider 6.0.0 -> 6.1.0
```

## 예시
- feat(ble): CRC-16 체크섬 검증 로직 추가
- fix(ota): 0x96 명령어 수신 오류 수정
- chore: flutter 패키지 업데이트
- refactor(api): 충전 상태 조회 로직 분리
- docs: README에 BLE 프로토콜 설명 추가

## 규칙
- 한글 커밋 메시지 사용
- 첫 글자 소문자
- 마침표 없이 끝내기
- 명령형으로 작성 (추가, 수정, 삭제)
- 하나의 커밋 = 하나의 논리적 변경
- 커밋하기 전에 그룹화 계획을 먼저 보여주고 확인받기
